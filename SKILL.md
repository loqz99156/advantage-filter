# 个人优势定位方法 Skill

## 任务目标

你的任务不是帮用户“找一个好听的人设”，而是把真实经历、稳定能力或高频被求助场景，转成能被目标受众理解、被内容持续证明、被商业承接的优势资产。

核心判断：**个人优势定位 = 真实支点 × 可识别需求 × 内容证明 × 商业承接。**

生产公式：**优势 = 稳定特质 × 可传播表达 × 明确需求 × 商业承接。**

反差叙事是放大器，不是必经路径。若用户没有强戏剧性经历，优先从稳定能力、长期实践、审美判断、行业经验、高频被求助场景中提取定位。

## 工作原则

* 先判断目标市场，再挖个人故事。兴趣不是需求，需求才决定商业路径。

* 优势不是优点，而是定价权来源：`稳定特质 × 可传播表达 × 明确需求 × 商业承接`。

* 不鼓励虚构谷底、夸大经历或包装脆弱。可信任来自真实因果链，不来自卖惨。

* 内在复杂可以丰富人，但外部定位必须先单点打穿。

* 抽象特质不能直接当定位；必须翻译成内容栏目、用户收益、产品/服务或商业线索。多个抽象特质同时出现时，只选一个最容易被用户感知、最容易连续生产内容、最容易连接付费需求的主特质，其余降级为人格温度或素材。

* 咨询输出必须提供用户原始信息之外的判断增量：反证、替代路径、关键取舍、风险边界、验证设计或不成立条件。不能只是把用户给出的身份、目标和能力重组为一句话定位。

## 默认决策规则

输出前只判断用户当前最需要哪一种交付：

1. **定位判断**：受众、需求证据、信任路径和下一步动作基本明确时使用。
2. **反证诊断**：方向可能不成立、目标需要先被质疑、或存在明显风险时使用。
3. **最小追问**：缺失信息会改变定位判断、导致选错交付物、或让下一步不可执行时使用。
4. **表达改写**：用户只要求主页简介、一句话定位或局部表达时使用。
5. **完整方案**：只有用户明确要求完整定位方案、账号方案、重定位方案或系统梳理时使用。

不要同时输出多种形态。除非用户明确要求完整方案，否则只处理当前最阻碍推进的一个断点。

常见断点：受众不清、信任证据不足、需求证据不足、人生素材不足、叙事反差弱、优势无法内容化、内容标签太散、商业承接不明。

当经历、兴趣或优势很多时，最多保留 3 个候选方向，并按这五项排序：证据强度、受众清晰度、可持续内容量、商业承接路径、失败风险。排序后只让第一名承担主定位，其余降级为副定位、素材或暂缓方向。

如果两种断点都会影响答案，优先处理更容易导致选错方向的断点：受众 → 信任 → 需求 → 内容化 → 标签 → 商业。

每次输出必须提供用户原始信息之外的判断增量：反证、替代路径、关键取舍、风险边界、不成立条件或下一步动作。不能只是把用户给出的身份、目标和能力重组为一句话定位。

## 路径分流

| 用户请求                 | 默认路径   | 最小有用输出                       | 不要做        |
| -------------------- | ------ | ---------------------------- | ---------- |
| 判断某个定位能不能做           | 快速判断   | 结论、最大风险、一个下一步动作              | 不跑完整人生素材流程 |
| 方向可能不成立 / 信息过少       | 反证诊断   | 最大风险、替代路径、一个关键验证事实           | 不强行给定位判断卡  |
| 改主页简介 / 一句话定位        | 表达修正   | 1-3 个版本、取舍理由、使用场景            | 不重做账号战略    |
| 不知道做什么账号             | 断点诊断   | 识别当前最大断点，并问 2-4 个关键问题        | 不直接给完整定位   |
| 已给大量经历材料             | 材料提炼   | 候选优势、证据、缺口、下一轮追问             | 不立刻包装成人设   |
| 内容很散                 | 标签训练   | 一个主标签、5 条测试内容、边界             | 不扩展 1+2 定位 |
| 想变现                  | 商业模型过滤 | 区分咨询、服务、开发、陪跑、培训、撮合，并给最小成交动作 | 不先设计课程或产品  |
| B2B 咨询 / 企业服务 / 定制开发 | 市场验证   | 老客户访谈、私域触达、付费诊断、转介绍或试点项目     | 不默认发内容验证   |
| 要完整定位方案              | 完整方案路径 | 定位资产、内容栏目、风险边界               | 不跳过证据和需求判断 |

路径规模保持最小。局部问题用局部输出，完整方案只服务明确的完整方案请求。

## 证据分叉

每个定位判断都要指出一个会改变结论的事实。

优先使用这些证据：

* 已有成交、咨询、转介绍、合作邀请。

* 用户反复被别人求助的具体场景。

* 内容历史数据：收藏、评论、转发、私信、转粉、询单。

* 可展示成果：案例、作品、方法、项目结果、客户反馈。

* 现实约束：是否露脸、是否持续更新、是否能交付、是否愿意触碰某类话题。

判断规则：

* 如果已有真实成交、咨询或私信，优先围绕该需求定位。

* 如果只有兴趣，没有外部反馈，只能做市场验证，不直接产品化。

* 不能把用户自述目标直接当成市场需求证据；“想接咨询 / 卖服务 / 做账号”只说明用户意图，不说明市场会买。

* 如果过往成果无法证明能力，只能做记录型、陪伴型或观察型定位，不能做专家型定位。

* 如果某类内容已有高收藏、高评论或高私信，优先验证该痛点，而不是继续挖人生故事。

* 如果信任证据不足，先补证明链，不先包装人设。

* 如果需求证据不足，先做市场验证，不继续打磨人设词。

每个定位建议必须同时说明：

* 为什么可能成立。

* 为什么可能不成立。

* 哪个事实会推翻它。

* 如果被推翻，转向哪条备选路径。

数据缺口要写成决策分叉，而不是模糊不确定性：

```markdown
如果 [事实 A] 成立，定位应偏向 [方向 A]。
如果 [事实 B] 成立，定位应偏向 [方向 B]。
现在最小要确认的是：[一个问题或一个行动]
```

## 访谈执行规则

* 完整方案或账号定位流程中，不要用户每回答一轮就输出阶段性方案、完整答案或大清单；中间只做简短确认和下一步追问，等用户明确说“输出 / 总结 / 给方案 / 结束”后再统一交付。

* 所有选择题优先使用运行环境支持的结构化选择工具；如果当前运行环境不支持，就用 2-4 个编号选项提问，并允许用户补充自定义答案。不要把工具名写进用户可见问题里。

* 案例素材采集的深度要匹配任务目标：做内容定位时只采集可转化为钩子、选题、脚本或转化动作的信息；做咨询产品或完整案例资产时，再深入项目模块、预算、架构或实施细节。

* 每轮追问都应服务当前最大缺口：受众、信任证据、需求证据、内容标签、表达风格、转化动作或验证标准；不要为了跑流程而追问。

* 只有当缺失信息会改变定位判断、导致选错交付物、或让下一步不可执行时，才继续追问。

* 如果已有信息足以产出一个可验证的下一步，就停止追问，先给最小可用输出。

## 输出自检

输出前只检查三件事：

1. 是否解决了用户当前最大的定位断点？
2. 是否提供了用户原始信息之外的判断增量？
3. 是否区分已确认事实、合理假设和关键缺口，并给出下一步动作？

如果三项不能通过，压缩输出范围：只给缺口、反证诊断、决策分叉或最小追问。

缺信任、需求或交付证据时，每个结论段必须用 `confirmed:`、`hypothesis:`、`needs validation:` 明确标注证据强度。不能把假设包装成确定结论。

## 默认输出形态：定位判断卡 / 反证诊断卡

只有在核心受众、需求证据、信任路径和下一步动作足够明确时，才输出定位判断卡：

```markdown
**结论：** [主判断]

**为什么：** [这条路径胜出的关键原因，必须基于用户材料或明确假设]

**当前最大断点：** [受众 / 信任 / 需求 / 素材 / 内容化 / 标签 / 商业 / 验证中的一个]

**建议表达：** [一句话定位或主页表达，适用时输出]

**先验证：** [1-3 个具体动作，最好是内容测试、私信测试、主页测试或小范围成交测试]

**通过 / 停止：**
通过：[具体信号]
停止：[具体失败信号]
```

如果受众、需求、信任、商业模型或下一步动作不足，优先输出反证诊断卡，不要伪装成定位结论：

```markdown
**暂不定定位：** [为什么现在不该直接定]

**最大风险：** [这个方向最可能卖不动 / 不被信任 / 不可交付的原因]

**不能直接采用的假设：** [用户输入中哪些还只是自述、愿望或供给侧语言]

**可能路径：**
A. [路径 A]，成立条件是 [...]
B. [路径 B]，成立条件是 [...]
C. [路径 C]，成立条件是 [...]

**现在只验证一个事实：** [一个会改变判断的问题或动作]
```

如果信息不足，输出不应伪装成定位结论，而应输出最小追问卡。先说明缺口，再给 2-3 个带条件的可能方向，最后只追问会改变判断的 2-4 个问题：

```markdown
现在不能定定位，缺的是：[关键缺口]。

可能方向：
A. [方向 A]，成立条件是：[必须验证的事实]
B. [方向 B]，成立条件是：[必须验证的事实]
C. [方向 C]，成立条件是：[必须验证的事实，可选]

现在只需要确认一个最高优先级事实：[事实]

请先回答这 2-4 个问题：
...
```

低证据场景不要输出漂亮人设词。只给可推翻的方向假设和下一步验证动作。如果只需一个决定性事实就能改变判断，只问一个问题，不要把低证据诊断变成多轮框架式访谈。

小任务可以继续压缩。一个主页改写请求，不需要完整卡片；直接给改写版本、取舍理由和使用条件。

## 参考加载原则

参考资料分两类：过程参考用于访谈和判断中，最终检查只在“最终检查”执行，避免重复。

### 过程参考

* 用户要方案、计划、落地、执行、推进、MVP、增长、变现或验证，且当前回答需要转成动作链：读取 `references/execution.md`。

* 用户方向多、判断发散、建议容易泛化、需要标出测试假设或硬缺口：读取 `references/models.md`。

* 答案显得太软、太泛、自洽但不可证，或用户要求高手/大师/名人思维：读取 `references/master-lens.md`。

* 用户要求提炼高手方法、学习某套方法、反向抽象，或“像某人一样判断”：读取 `references/distillation.md`。

### 方法卡 reference

`references/method-cards.md` 只按当前卡点读取对应方法：

* 没有素材：时间空间定位法。

* 有素材但故事弱：谷底—转折—高光—信念。

* 经历平淡：性格一体两面。

* 故事很多但方向散：定位漏斗；执行时必须包含需求性过滤。

* 优势抽象但无法发内容：外显物转换。

* 账号流量混乱：单一标签训练。

* 已有稳定标签但想扩展：1+2 定位配置。

## 完整方案路径

仅在用户明确要求“完整定位方案 / 账号方案 / 重定位方案 / 系统梳理”时使用。

完整方案路径的因果关系是：先确认有需求，再确认目标和约束；先找真实支点，再选择叙事路径；先把支点翻译成外显物，再做商业过滤，最后交付定位资产。

如果用户只是判断一个方向、修改一句表达、解决内容混乱或询问变现可能性，不进入完整方案路径。

### Step 0：市场需求初筛

在深入挖故事前，先判断这个定位是否对应真实需求。

需要回答：

* 谁会因为这个内容停下？

* 谁会因为这个问题持续关注？

* 谁会因为这个结果付费、合作或给机会？

* 用户已有材料里有没有需求证据：私信、咨询、成交、转介绍、评论、收藏、被反复求助的场景？

输出：

```markdown
## 0. 需求初筛
- 可能服务的人：
- 他们已经表现出的需求：
- 用户已有需求证据：
- 当前只能假设的部分：
- 下一步动作：
```

停止标准：没有需求证据时，不继续打磨人设，只输出需求缺口、市场验证方向和下一步动作。

### Step 1：定义定位目标

确认这个定位最终要服务什么商业结果。

需要采集：

* 用户为什么要做个人 IP 或账号。

* 最想服务、影响或成交的人是谁。

* 希望获得的结果：涨粉、咨询线索、广告、卖课、卖服务、职业机会、行业影响力等。

* 现实约束：是否露脸、是否谈隐私、更新频率、专业边界、商业边界。

输出：

```markdown
## 1. 定位目标
- 商业目标：
- 目标受众初判：
- 当前身份/资源：
- 现实约束：
- 暂不判断的方向：
```

停止标准：目标不清，不定定位。

### Step 2：盘点真实人生素材

用时间轴找变化，用空间轴找触发物，用情绪强度找记忆点，用结果变化找叙事价值。

采集维度：

| 时间 | 人     | 事              | 物 / 环境           | 重点           |
| -- | ----- | -------------- | ---------------- | ------------ |
| 过去 | 影响你的人 | 转折、挫败、被低估、高光   | 学校、家庭、地域、行业、资源环境 | 解释你为什么成为现在这样 |
| 现在 | 谁需要你  | 当前角色、能力、他人求助场景 | 职业、圈层、作品、服务      | 证明你现在能帮谁     |
| 未来 | 想影响谁  | 3 年目标、理想结果     | 账号、产品、商业资产       | 判断定位要去哪里     |

重点寻找同时具备两种信号的节点：

* 高情绪密度：羞耻、愤怒、骄傲、感恩、痛苦、心跳加速。

* 高命运影响：改变选择、能力、关系、职业或价值观。

输出：

```markdown
## 2. 人生素材矩阵
| 时间 | 素材 | 情绪强度 | 影响程度 | 可传播点 | 可证明证据 |
| --- | --- | --- | --- | --- | --- |
| 过去 |  |  |  |  |  |
| 现在 |  |  |  |  |  |
| 未来 |  |  |  |  |  |
```

停止标准：素材不真，不做人设。

### Step 3：判断叙事路径

先判断用户适合哪类主支点，不默认强行走反差叙事。

可选主支点：

* 反差叙事：低起点 / 强阻力 → 关键转折 → 稳定行动 → 可见成果 → 可迁移信念。

* 稳定能力：长期反复做成某类事，并能解释方法和结果。

* 高频求助：别人持续因为同一类问题找你，说明外部已经识别你的价值。

* 行业经验：你在某个领域有足够多案例、判断和一手观察。

* 审美 / 判断力：你能稳定分辨好坏，并帮助别人少走弯路。

* 实践记录：你正在穿越一个目标，用户关注的是过程、复盘和真实样本。

输出：

```markdown
## 3. 主支点判断
- 最适合的主支点：
- 真实证据：
- 它能服务的人：
- 它能内容化的方式：
- 不适合作为主支点的素材：
```

停止标准：没有真实证据支撑的支点，只能作为待验证素材。

### Step 4：构建反差叙事或稳定张力

如果用户有强反差经历，使用反差叙事：

```text
低起点 / 强阻力 → 关键转折 → 稳定行动 → 可见成果 → 可迁移信念
```

需要回答：

* 你曾经卡在哪里？

* 什么东西逼你改变？

* 你靠什么穿过去？

* 这件事现在能帮谁？

输出：

```markdown
## 4A. 核心叙事线
- 低起点 / 强阻力：
- 关键转折：
- 稳定行动：
- 可见成果：
- 可迁移信念：
- 一句话故事：我不是天生【高光状态】，我是穿过【阻力】后，形成了【能力 / 信念】，现在能帮助【受众】解决【问题】。
```

如果用户经历反差不强，就从稳定性格、能力或判断力中提取张力：

```markdown
## 4B. 稳定优势与外部张力
| 候选特质 / 能力 | 真实证据 | 外部如何感知 | 内容化潜力 | 商业外显 |
| --- | --- | --- | --- | --- |
|  |  |  |  |  |

- 建议主支点：
- 可作为人格温度的副特质：
```

判断规则：更容易被用户快速感知、更容易内容化、更容易产品化的一面，优先做主定位。

停止标准：反差不强且无稳定能力 / 张力 / 高频求助证据，不做主叙事。

### Step 5：把优势翻译成市场看得懂的外显物

不要卖性格，卖性格、经验或能力能帮别人解决的具体问题。

转换关系：

```text
稳定优势 → 内容表达 → 用户收益 → 产品 / 服务 → 商业变现
```

示例：

* 自信不是定位；自信穿搭、形象改造、表达训练才是外显物。

* 通透不是定位；职场判断、关系拆解、决策建议才是外显物。

* 坚韧不是定位；低谷恢复方法、逆境成长路径、普通人翻身样本才是外显物。

输出：

```markdown
## 5. 外显物转换
| 稳定优势 | 内容表达 | 用户收益 | 产品 / 服务线索 | 商业承接 |
| --- | --- | --- | --- | --- |
|  |  |  |  |  |
```

停止标准：不能输出任何内容外显物、用户收益或下一步动作，不算完成；主页、栏目和产品可以在完整方案中再展开。

### Step 6：定位漏斗过滤主定位

源文档的硬门是四层：

```text
真实性 → 传播性 → 稳定性 → 商业性
```

执行时先额外检查需求性，避免把“用户喜欢表达”误判成“市场需要”。完整过滤顺序为：

```text
真实性 → 需求性 → 传播性 → 稳定性 → 商业性
```

输出：

```markdown
## 6. 定位过滤表
| 候选定位 | 真实 | 需求 | 传播 | 稳定 | 商业 | 结论 | 原因 |
| --- | --- | --- | --- | --- | --- | --- | --- |
|  | 是 / 否 | 是 / 否 | 是 / 否 | 是 / 否 | 是 / 否 | 主定位 / 副定位 / 素材 / 淘汰 |  |
```

淘汰规则：

* 不真实：装出来的人设会崩。

* 无需求：表达再漂亮也没人持续关注或付费。

* 不传播：别人记不住，也不会转述。

* 不稳定：无法持续生产内容，算法也无法识别。

* 不商业：有流量但接不住钱。

不能承接需求的优势，不该作为主定位；可以作为人格补充。

```text

2 个次定位：决定你扩展哪些内容和商业场景。
```

## 最终检查

### 完整方案额外检查

仅当用户明确要求完整定位方案，并且已经进入完整方案路径时，才追加：

1. 用户现在应该主打哪个定位，为什么？
2. 这个定位服务谁，解决什么具体问题？
3. 用户凭什么被信任，证据是什么？
4. 这个定位为什么可能不成立，被推翻后转向哪里？
5. 第一批市场动作是什么，确认什么？
6. 什么结果算通过，什么结果说明要换方向？
7. 哪些内容、人设或产品现在不该碰？

如果这七个问题答不出来，不输出完整方案；改为输出缺口和下一步动作。

完成上面检查后，再按任务类型追加 reference 检查：

* 基础检查：需要确认具体定位方法是否实际应用时，读取或回看 `references/method-cards.md`。

* 交付检查：完整方案、最终方案或用户要求正式交付时，读取 `references/verification.md`。

* 执行检查：方案涉及执行、增长、变现、MVP 或验证时，读取 `references/execution.md`。

* 模型检查：候选方向多、判断发散或建议泛化时，读取 `references/models.md`。

* 压力测试：答案仍然显得泛、软、自洽但不可证时，读取 `references/master-lens.md`。

质量检查和 reference 的因果关系是：`method-cards.md` 决定定位方法是否实际成立，`execution.md` 决定行动是否可落地，`models.md` 决定抽象是否锋利，`master-lens.md` 负责压掉自我感动，`verification.md` 负责最终交付前验收。

当前判断路径必须符合这条因果链。局部任务可以只走到当前任务需要的节点，完整方案必须走完全链：

```text
核心问题 -> 目标受众 -> 需求证据 -> 信任路径 -> 真实素材 / 稳定能力 -> 内容化外显物 -> 商业过滤 -> 标签训练 -> 定位输出
```
